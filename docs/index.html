Write your learning goal and puzzle trick

Before we dive into building a webpage, let's talk about what makes a good challenge: all good challenges involve a puzzle.

Puzzles give us a sense of play, imply a sense of control over an environment and focus our attention to solve a logical problem. When you figure out how to solve a puzzle, you've essentially learnt something. This is why we're making puzzles in our challenges.

A good puzzle needs you to learn something in order to solve it. So we will need two things to get started: a learning goal and a puzzle trick.

Learning goal What you want the students to learn by solving the puzzle.
Puzzle Trick The logical problem students need to figure out to solve a logical problem. e.g. the algorithms to solve a rubix cube depending on the current configuration.

Using those definitions as a guide, write the learning goal and puzzle trick for your puzzle. If you're stuck, check out this video where Scott Kim, Puzzlemaster, talks about different examples of puzzles.

Below is an example learning goal and puzzle trick for a challenge on Transforming Shapes. We'll be using this challenge as an example throughout the recipe. You can find it here for reference.

Learning goal Understand how translate and rotate affect how shapes are drawn on the canvas.
Puzzle Trick Together rotate() and translate() position and orientate the canvas to any part of the screen. You draw one blade of the windmill, set the origin to the middle of the screen and then use rotate() to iterate the canvas orientation every time a for loop runs

Now go and write your own learning goal and puzzle trick.

Work out your algorithm

We've just sketched out what our puzzle will look like. But before we can figure out how the student will solve the puzzle, we need to figure how the code will work for ourselves!

It might help to break down how it's going to work in a diagram before you go and develop it. Before you can teach this, you have to have working code for your solved puzzle. After you've got it working we can break it down into learnable steps.

Using your sketch as a guide, work out how the instructions you give the computer in p5.js will create your solved puzzle. This is to break down the code process and keep track of the steps you take to get to the solved puzzle.

You can do this is by writing a code process flow. Below is an example of a code process flow for our example challenge on transforming shapes.

This is a process flow for our windmill challenge

This first box says we need to 'make a blade' and we think the best method to use would be beginShape(). It's also mentioned we might need to put this inside a custom function for later.

The second box says we need to 'make 6 blades' and we think a for loop will be the easiest way to do that. It also says we need to iterate the blade code 6 times to make enough blades for our windmill.

The third box says we need to 'rotate 6 blades' and we think adding an 'r' argument to our custom function will make it easy to rotate each blade the right amount by iterating the 'r' value in a for loop and then using 'r' as the value in the rotate method.

The fourth box says we need to 'translate 6 blades' and we think adding 'x' and 'y' arguments to the custom function will, like with 'r', make it easy to move each blade on the screen the right amount. This time, instead of the rotate method, we'll put 'x' and 'y' in the translate() method.

The final box says we have an extra challenge 'blow on blades'. We think the p5.audioIn method would be the way to hook up the windmill to our mic. We've also said there needs to be a blownBreeze variable and a gentleBreeze variable to mimic how regular wind and someone blowing on a windmill (if they had enough breath!) would combine to make the windmill move. Then, it says the blownBreeze variable would be the mic amplitude and the gentleBreeze variable would use a counter to remain steady.

Design your puzzle pieces

So you’ve just written the code for your puzzle like the fantastic puzzle designer you are! Now you need to break it down into learnable steps so the student can learn something by solving the puzzle.

The way you break down these steps won't be the same as how you worked out your algorithm. You wrote that for someone who already knew how to code it from scratch i.e. you!

Now you're writing it for a less experienced student; someone who doesn't understand the code yet but will learn by solving your puzzle. Think what kind of puzzle pieces you’d want to start with if you were doing the challenge as a Design Programming student. Always keep your target user in mind.

Don't worry if this is confusing. Human Centred Design is like that for everyone at the start!

Write your challenge instructions

Now you’ve figured out what your puzzle pieces are, it's time to write the instructions you'll give your students so they understand what they need to do to complete their challenge.

Remember, this isn't just giving your student the puzzle and telling them to solve it. You need to make sure that the understand what they need to do at each section of your challenge.

You need to help them understand what they need to do without giving the puzzle away.

To do this write out your instructions in a word document. See if you can mimic the writing style in the example challenge i.e. splitting the challenge into sections, using titles for sections and providing a clear action for each section with support visual references.
